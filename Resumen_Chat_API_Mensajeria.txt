============================================================
2. CORRECCIONES REALIZADAS
============================================================

CORRECCIÓN 1: CONEXIÓN A SQL SERVER

Problema:
La API no podía conectarse a SQL Server.

Causa:
La cadena de conexión apuntaba a una instancia incorrecta o no disponible.

Solución:
Se recomendó revisar el nombre real de la instancia en SQL Server Management Studio y ajustar appsettings.json.

Ejemplos:
Server=localhost;
Server=.\SQLEXPRESS;
Server=KAZ\SQLEXPRESS;
Server=(localdb)\MSSQLLocalDB;


CORRECCIÓN 2: ERROR DE SWAGGER Y OPENAPI

Problema:
Aparecieron errores con:

Microsoft.OpenApi.Models
OpenApiInfo

Causa:
La versión instalada de Swashbuckle usaba cambios nuevos en namespaces.

Solución:
Se recomendó usar Swashbuckle.AspNetCore versión 6.6.2 para mantener compatibilidad con:

using Microsoft.OpenApi.Models;


CORRECCIÓN 3: CAMBIO DE INTERFAZ GRÁFICA A API PURA

Problema:
Primero se había creado una interfaz con Tailwind, pero después ya no era necesaria.

Solución:
Se decidió eliminar:
- Pages.
- wwwroot.
- Archivos visuales.

Se dejó la API funcionando únicamente con:
- Controllers.
- Models.
- DTOs.
- Services.
- Data.
- Swagger.


CORRECCIÓN 4: VALIDACIÓN DE ESTADOS

Problema:
La validación de estados estaba escrita directamente en C# usando textos.

Limitación:
Podía funcionar, pero no era lo más ordenado para un proyecto con base de datos.

Solución:
Se movió la lógica de estados a SQL Server.

Se agregaron:
- EstadoEnvio.
- TransicionEstado.
- IdEstado en Envio.
- IdEstadoNuevo en HistorialEnvio.

Ahora la API consulta SQL para saber si una transición es válida.


CORRECCIÓN 5: JSON DE CAMBIO DE ESTADO

Antes se enviaba:

{
  "nuevoEstado": "EnTransito",
  "idOficina": 1,
  "notas": "Cambio de estado"
}

Después se cambió a:

{
  "idEstadoNuevo": 2,
  "idOficina": 1,
  "notas": "Cambio de estado"
}

Esto se hizo porque los estados ahora se manejan por ID desde la base de datos.


CORRECCIÓN 6: REGLAS DE NEGOCIO PARA PRUEBAS UNITARIAS

Problema:
Las reglas de tarifa, NIT e intentos estaban dentro del servicio, lo que hacía más incómodo probarlas directamente.

Solución:
Se creó una clase separada:

Rules/EnvioRules.cs

Con métodos para:
- CalcularTarifa.
- NitEsValido.
- CalcularDescuento.
- CalcularTotal.
- CalcularNuevoIntento.
- DebePasarADevolucion.

Esto facilitó crear pruebas unitarias limpias.

1. Listar oficinas.
2. Listar estados.

10. Aplicar descuento si hay NIT válido.

12. Bloquear estados inválidos.
13. Cambiar automáticamente a EnDevolucion al tercer intento fallido.


============================================================
5. FLUJO DE ESTADOS
============================================================

Estados:

1 = Registrado
2 = EnTransito
3 = EnReparto
4 = Entregado
5 = EnDevolucion
6 = Devuelto

Transiciones permitidas:

Registrado -> EnTransito
EnTransito -> EnReparto
EnReparto -> Entregado
EnReparto -> EnDevolucion
EnDevolucion -> Devuelto

Ejemplo de flujo normal:

Registrado -> EnTransito -> EnReparto -> Entregado

Ejemplo de flujo con devolución:

Registrado -> EnTransito -> EnReparto -> EnDevolucion -> Devuelto

La API no permite regresar a un estado anterior porque valida el cambio en la tabla TransicionEstado.

8. REFLEXIÓN DEL PROCESO
============================================================

En este chat se empezó creando una API REST con una interfaz visual, pero durante el desarrollo se fue ajustando según lo que realmente se necesitaba.

Primero se pensó en una interfaz con Tailwind para registrar y rastrear paquetes. Después se mejoró la interfaz para que fuera más dinámica y se dividió en varias páginas. Más adelante se decidió que no era necesaria la parte gráfica y que era mejor trabajar solo con Swagger, porque así era más directo probar la API.

También se corrigieron errores importantes, como el problema de conexión con SQL Server y el problema de Swagger con Microsoft.OpenApi.Models. Esos errores ayudaron a limpiar mejor la configuración del proyecto.

Una mejora importante fue cambiar la lógica de estados. Al principio se validaba desde C# usando texto, pero después se decidió hacerlo mejor con tablas en SQL. Eso dejó la API más ordenada, porque ahora los estados tienen ID y las transiciones válidas están en la base de datos.

También se agregaron pruebas unitarias con xUnit. Para eso se separaron reglas como tarifa, NIT, descuento e intentos en una clase aparte. Esto hizo que el proyecto quedara más fácil de probar y defender.

En general, el proyecto fue evolucionando desde una idea visual hasta una API más limpia, probada con Swagger, con base de datos organizada y con pruebas unitarias.
